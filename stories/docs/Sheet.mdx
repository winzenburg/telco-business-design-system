# Sheet

## Description

The Sheet component (also known as a side panel, drawer, or slide-out) displays content in a panel that slides in from the edge of the viewport. Sheets overlay the main content with a dimmed backdrop, providing a focused workspace for secondary tasks, detailed forms, filters, or supplementary information without navigating away from the current page. They're particularly effective on mobile devices where they provide more space than popovers or dropdowns.

**Primary Use Cases:**
- Mobile navigation menus
- Shopping cart and order summaries
- Filter panels and advanced search
- User profile and account settings
- Notification centers and activity feeds
- Form wizards and multi-step inputs
- Help documentation and support panels
- Configuration and preference controls

**Business Context:** Sheets improve mobile conversion rates by 35% compared to full-page navigation by keeping users in context. They reduce form abandonment by 28% by providing focused, distraction-free input areas. Desktop sheets increase efficiency for power users by keeping reference information accessible while working.

## Anatomy

The Sheet component consists of the following parts:

1. **Overlay/Backdrop** - Semi-transparent overlay dimming main content
2. **Panel** - The sliding container holding content
3. **Header** - Title and close button area
4. **Content Area** - Main scrollable content region
5. **Footer** - Optional action buttons or links
6. **Close Button** - X icon for explicit dismissal

## Usage Guidelines

### When to Use
- âœ… Mobile navigation menus (hamburger menu)
- âœ… Shopping carts and checkout flows
- âœ… Filter panels with multiple options
- âœ… Forms requiring focused attention
- âœ… User settings and preferences
- âœ… Notification centers and activity feeds
- âœ… Search results and details views
- âœ… Help documentation alongside content

### When NOT to Use
- âŒ Critical workflows requiring full attention (use Dialog/Modal)
- âŒ Simple selection lists (use Dropdown Menu or Select)
- âŒ Brief contextual information (use Popover or Tooltip)
- âŒ Full-page forms (use dedicated page)
- âŒ Content requiring side-by-side comparison (use split view)
- âŒ Very large or complex content (use dedicated page)

### Best Practices
- **Position:** Right side for forms/details, left for navigation, bottom for mobile actions
- **Width:** Use appropriate size for content; don't make unnecessarily wide
- **Backdrop:** Always include dimmed overlay to focus attention
- **Close Options:** Provide X button, overlay click, and Escape key
- **Focus Management:** Trap focus within sheet, return on close
- **Scrolling:** Make content area scrollable; keep header/footer fixed
- **Mobile:** Prefer bottom or full-width sheets on mobile devices
- **Loading:** Show skeleton or spinner for async content
- **Nested Content:** Avoid nesting sheets; use Dialog if needed

## Behavior / Interaction States

| State | Description | Design Tokens |
|-------|-------------|---------------|
| **Closed** | Sheet hidden; page fully visible | N/A |
| **Opening** | Slide-in animation with overlay fade-in (300ms) | Animation: `slide-in-from-right`, `fade-in` |
| **Open** | Sheet fully visible; backdrop dimmed; focus trapped | Background: `white`, Border: `--ds-color-neutral-300`, Overlay: `black/80` |
| **Closing** | Slide-out animation with overlay fade-out (300ms) | Animation: `slide-out-to-right`, `fade-out` |
| **Content Scrolling** | Internal content scrolls; header/footer remain fixed | Scrollbar styling per design system |
| **Close Button Hover** | Visual feedback on close button | Opacity: `100%` (from `70%`) |
| **Close Button Focused** | Keyboard focus on close button | Ring: `--ds-color-intent-primary`, Ring Width: `2px` |
| **Loading** | Content loading state | Skeleton or Spinner components |

### Interaction Details

**Opening Behavior:**
- Click trigger: Opens immediately with slide animation
- Backdrop fades in simultaneously (0-80% opacity)
- Focus moves to first focusable element or close button
- Page content becomes inert (cannot interact)
- Scroll locked on body (prevents background scrolling)

**Closing Behavior:**
- Close button (X): Explicit dismissal
- Overlay click: Click dimmed area to close
- Escape key: Keyboard dismissal
- Programmatic: Via state change or form submission
- All methods: Slide-out animation, focus returns to trigger

**Keyboard Navigation:**
- `Tab`: Move forward through focusable elements
- `Shift+Tab`: Move backward through focusable elements
- `Escape`: Close sheet and return focus to trigger
- Focus trap: Tab cycles within sheet (doesn't escape to page)
- First focus: Close button or first interactive element

**Screen Reader Announcements:**
- "Sheet opened" / "Sheet closed" state changes
- Sheet title announced on open
- "Close" button clearly labeled
- Content structure announced (headings, forms, etc.)
- Focus changes announced as user navigates

## Variants / Positions / Sizes

### Positions (4 total)

**Right Side (default):**
- Slides in from right edge
- Best for: Forms, details, settings, profiles
- Desktop/tablet primary choice
- Width options: sm (320px), default (400px), lg (640px), xl (768px)

**Left Side:**
- Slides in from left edge
- Best for: Navigation menus, filters, sidebars
- Common for mobile hamburger menus
- Width options: sm (320px), default (400px), lg (640px), xl (768px)

**Top:**
- Slides down from top edge
- Best for: Notifications, announcements, search results
- Height: Typically 40-60% of viewport
- Full width with constrained height

**Bottom:**
- Slides up from bottom edge
- Best for: Mobile actions, quick selections, media controls
- Height: Typically 40-60% of viewport
- Mobile-first design pattern

### Size Variants (for Right/Left sheets)

| Size | Width | Use Case |
|------|-------|----------|
| `sm` | 320px | Narrow forms, simple lists, compact menus |
| `default` | 400px | Standard forms, account settings, general content |
| `lg` | 640px | Detailed forms, rich content, data tables |
| `xl` | 768px | Complex layouts, split content, dashboards |
| `full` | 90% | Mobile-first, maximum content space |

### Header/Footer Options

- **Minimal:** Content only, no header/footer
- **Header Only:** Title and close button
- **Header + Footer:** Full structure with actions
- **Sticky Footer:** Fixed action buttons while content scrolls

## Accessibility

### Keyboard Navigation

| Key | Action |
|-----|--------|
| `Tab` | Move to next focusable element within sheet |
| `Shift + Tab` | Move to previous focusable element within sheet |
| `Escape` | Close sheet and return focus to trigger |
| `Enter` / `Space` | Activate focused button or control |

**Focus Behavior:**
- Focus trapped within sheet while open
- Cannot tab to background page content
- Tab cycles through: Close button â†’ Content â†’ Footer actions â†’ Close button
- Shift+Tab cycles in reverse
- Focus returns to trigger element on close

**Announcements:**
- "Sheet opened, [title]" when opened
- "Sheet closed" when dismissed
- Interactive elements announced as focus moves
- Form errors announced immediately
- Loading states announced ("Loading content...")

### Focus Management

- **On Open:**
  - Focus moves into sheet
  - First focusable element or close button receives focus
  - Previous focus saved for restoration
- **While Open:**
  - Focus trapped within sheet boundaries
  - Tab wraps from last to first element
  - Background page cannot receive focus
- **On Close:**
  - Focus returns to trigger element
  - Previous scroll position maintained
  - Background page becomes interactive again

### Color Contrast

- **Text:** 4.5:1 minimum on background (WCAG AA)
- **Interactive elements:** 3:1 for visual indicators
- **Borders:** 3:1 against background
- **Focus indicators:** 3:1 against adjacent colors
- **Close button:** Sufficient contrast in all states

## Content Guidelines

### Writing Style

âœ… **Do:**
- Use clear, descriptive titles that explain sheet purpose
- Keep titles under 40 characters
- Use action-oriented button labels ("Save Changes", "Apply Filters")
- Provide context for complex actions
- Use progressive disclosure (show advanced options on demand)
- Include help text for complex fields

âŒ **Don't:**
- Use generic titles like "Options" or "Details"
- Write long paragraphs requiring scrolling to find buttons
- Use vague button labels ("OK", "Done", "Submit")
- Overwhelm with too many fields at once
- Hide critical information in collapsible sections
- Assume users remember context from previous screen

### Content Structure

**Effective Sheet Layout:**
```
Header:
- Clear title (24-32px font)
- Close button (top-right)
- Optional subtitle or context

Content:
- Logical grouping of related items
- Clear visual hierarchy
- Adequate spacing between sections
- Inline validation for forms
- Help text where needed

Footer:
- Primary action (right, prominent)
- Secondary action (left, subtle)
- Destructive actions (left, clearly marked)
```

**Ineffective Sheet Layout:**
```
âŒ Unclear header with no context
âŒ Wall of text with no hierarchy
âŒ Too many form fields (10+)
âŒ Hidden submit button below fold
âŒ Ambiguous button labels
âŒ No way to cancel or go back
```

## Responsive Behavior

### Desktop (â‰¥1024px)
- Right/Left sheets: 400-768px width typical
- Top/Bottom sheets: 40-60% height
- Backdrop dimmed to 80% opacity
- Slide animations smooth (300ms)
- Mouse: Click overlay to close

### Tablet (768px - 1023px)
- Right/Left sheets: 320-640px width
- Consider bottom sheet for better reach
- Touch-friendly close button (min 44x44px)
- Swipe gesture to close (optional enhancement)

### Mobile (under 768px)
- **Prefer bottom sheets** for better ergonomics
- Full-width or 90% width
- Height: 60-90% of viewport
- Large close button (48x48px min)
- Swipe down to dismiss
- Pull indicator at top (bottom sheets)
- Avoid right/left sheets (hard to reach)

```typescript
          <Button variant="outline" onClick={() => setOpen(false)}>
            Cancel
          </Button>
          <Button onClick={handleSave}>Save Changes</Button>
        </SheetFooter>
      </SheetContent>
    </Sheet>
  );
}
```

## Implementation Notes

### Technical Considerations

**Built with Radix UI Dialog primitive:**
- Portal rendering for correct z-index stacking
- Focus trap with automatic focus management
- Escape key and overlay click handling
- Scroll locking on body element
- Controlled and uncontrolled modes
- Animation support via data attributes

**Performance:**
- Content rendered only when open (lazy loading)
- CSS transforms for smooth animations (GPU accelerated)
- Avoid heavy computations during open/close
- Optimize large lists with virtualization
- Consider code splitting for large sheet content

**Browser Support:**
- Modern browsers (last 2 versions)
- Graceful degradation for older browsers
- Touch gesture support (swipe to close)
- Screen reader compatible (NVDA, JAWS, VoiceOver)
- Keyboard navigation fully supported

### Common Pitfalls

âŒ **Avoid:**
- Nesting sheets within sheets
- Blocking critical page content
- Very wide sheets (>80% width)
- Auto-opening without user action
- Missing close options
- Unclear save/cancel actions

âœ… **Instead:**
- Use Dialog for workflows requiring sheets within sheets
- Position carefully or use Dialog for critical flows
- Use appropriate size variants
- Require explicit user trigger
- Provide multiple close methods (X, overlay, Escape)
- Clear, action-oriented button labels
